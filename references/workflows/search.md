# 查找流程

> 用户遇到问题时，从已有记录中检索

## ⛔ 执行约束（强制）

- 禁止直接基于模型记忆回答“是否有历史经验”。
- 必须先在 `data/` 目录下执行本地检索（文件搜索/文本搜索），再决定是否网络搜索。
- 本地未命中时，必须先明确告知“阿卡西本地未命中”；除非用户明确要求继续调研，或意图本身就是外部调研，否则不得自动进入网络搜索。
- 引用结果前必须检查记录状态与适用版本，不可直接复述过期结论。
- 搜索范围仅限 `data/*.md` 记录；`SKILL.md` 与 `references/*` 仅为系统操作规范，禁止作为用户查询结果返回。
- 禁止将 `references/INDEX.md` 全文直接作为检索上下文发送给大模型。

### 搜索范围白名单（唯一有效）

- ✅ 允许搜索并返回：`data/*.md`
- ❌ 禁止作为查询结果返回：`SKILL.md`、`references/**`、`assets/**`、技能目录外文件
- 若仅在禁止路径中命中相关内容，必须判定为“本地未命中”，并根据用户意图决定是否继续执行网络搜索流程

---

## 一、查找步骤

1. **提取关键词** → 从用户问题中提取核心概念
2. **本地检索** → 使用 AI 可用搜索工具，仅在 `data/` 目录下按关键词/标签/文件名检索
3. **读取记录** → 读取命中的 `data/*.md` 对应文件
4. **呈现结果**：
   - ✅ 找到 → 展示内容，询问是否适用
   - ❌ 未找到 → 明确说明“阿卡西本地未命中”，再根据用户意图决定是否转入 **网络搜索**

---

## 二、搜索策略

### 本地检索（推荐）
使用 AI 可用搜索工具在 `data/` 目录下检索：
- 按关键词：在 `data/*.md` 中搜索核心术语（如“协程”“yield”“CBUFFER”）
- 按标签：在 `data/*.md` 元数据中搜索 `#unity`、`#shader` 等标签
- 按文件名：在 `data/` 下按主题关键词匹配文件名

> 说明：`references/INDEX.md` 仅用于索引维护，不作为检索语料来源。

### 搜索技巧

| 技巧 | 说明 |
|------|------|
| **目录约束** | 所有本地检索仅限 `data/`，避免系统层文件污染结果 |
| **标签过滤** | 在 `data/*.md` 中按 `#tag` 搜索，优先匹配标签行 |
| **关键词** | 使用核心概念而非整句，先窄后宽逐步扩展 |
| **状态优先** | 优先采纳 ✅ 已验证 / 📘 有效的记录 |
| **版本检查** | **关键**：确认记录版本与当前环境匹配 |

---

## 三、网络搜索（仅在本地未找到且需要继续调研时）

**步骤**：

1. **告知用户**："阿卡西本地未命中，正在继续做外部搜索..."
2. **构建搜索词** → 提取核心关键词
3. **执行搜索** → 使用可用的网络搜索工具

**搜索来源优先级**：

| 优先级 | 来源 |
|--------|------|
| 1 | 官方文档（Unity Docs、Microsoft Docs、MDN） |
| 2 | 官方论坛（Unity Forum、GitHub Discussions） |
| 3 | Stack Overflow（高票答案优先） |
| 4 | 技术博客（知名作者、详细解析） |

**结果评估**：

| 检查项 | 说明 |
|--------|------|
| 时效性 | 内容发布日期，API 是否已变更 |
| 权威性 | 官方 > 社区 > 个人 |
| 完整性 | 是否有完整示例代码 |

**后续**：解决问题后，如认为值得记录，向用户展示拟记录内容摘要并征求确认，用户同意后转入 [record.md](./record.md)

## 四、输出要求

- 明确区分“阿卡西本地记录结论”和“外部调研结论”。
- 若本地未命中但执行了外部调研，必须先说明“本地未命中”，不得把外部结论表述成已有库存量知识。
- 若用户只是在问“之前有没有记录过”，在本地未命中后可以止步，不必自动扩展为完整调研任务。
