# 治理流程

> 用于调整阿卡西系统层规则，确保模板、workflow、索引机制与 Web 解析契约保持一致。

## ⛔ 执行约束（强制）

- 仅当需求涉及模板、workflow、字段 schema、状态体系、索引规则、标签治理规则、Web 解析契约或系统层说明文档时，才进入本流程。
- 治理流程不得拿系统层文档充当用户知识回答来源；知识查询仍只允许返回 `data/*.md`。
- 在兼容性未评估清楚前，禁止直接修改正式记录结构。
- 若治理变更会影响 Web 渲染、索引脚本或历史记录解析，必须先列出影响范围，再实施修改。

## 一、适用场景

| 场景 | 示例 |
|------|------|
| 模板治理 | 调整记录模板、元数据字段、状态定义 |
| 流程治理 | 优化 search/record/validate 的边界与完成条件 |
| 索引治理 | 调整 INDEX、tag-registry 的维护规则 |
| 契约治理 | 修改 Web 解析依赖的结构、字段、路径约定 |
| 历史整治规划 | 制定全库结构修复、状态收敛、标签合并方案 |

## 二、执行步骤

1. **界定变更类型**
   - 明确本次是模板、workflow、索引、schema 还是 Web 契约治理。
2. **评估影响范围**
   - 列出会受影响的文件、脚本、Web 渲染逻辑和历史记录范围。
3. **确认兼容策略**
   - 判断是纯文档澄清、非破坏性约束增强，还是需要后续历史数据整治。
4. **实施系统层修改**
   - 仅修改必要的系统层文档或索引层规则，避免顺手改动无关内容。
5. **交叉核对**
   - 检查 SKILL、workflow、模板、INDEX、tag-registry 是否出现互相矛盾。
6. **说明后续动作**
   - 如存在存量数据待整治、Web 侧待同步、脚本待更新，必须明确列出，不得隐含带过。

## 三、强制测试步骤

在治理变更完成后，**必须**执行以下脚本测试：

```bash
python references/scripts/validate_records.py
python references/scripts/regenerate_index.py
python references/scripts/validate_records.py
```

- 若 `validate_records.py` 报告的是历史存量问题，必须明确区分“脚本功能正常”与“数据仍待整治”；禁止把历史脏数据误判为脚本失效。
- 若 `regenerate_index.py` 运行失败或引入新的 `INDEX.md` 差异异常，必须先修正脚本或规则，再继续后续工作；禁止带着失败测试进入汇报阶段。

## 四、兼容性检查

- 现有 `data/*.md` 是否仍能被模板与 Web 规则解释。
- 状态定义是否与历史记录、INDEX 展示和前端映射保持一致。
- 索引层文档是否仍与数据层、系统层描述一致。
- 是否引入了新的必填字段、结构变更或路径规则变化。
- 是否需要单独开展历史数据清洗，而不是在普通记录流程中零散修补。

## 五、输出要求

- 明确说明本次改的是“系统规则”而非“知识内容”。
- 对每项变更给出原因，重点解释它如何降低结构漂移和流程歧义。
- 若为了兼容性暂未落地某项结构改动，必须明确写出保留原因。
